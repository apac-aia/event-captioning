1. 一句話結論

本章重點

先確認混音器乾淨人聲訊號，再比較人工智慧工具；沒有乾淨聲音，後面都是猜。

ADALS 不該先問「哪個人工智慧字幕工具最好？」
ADALS 應該先問「我們能不能從現場混音器拿到一條乾淨、穩定、只包含人聲的音訊？」

這條乾淨音訊是整套系統的主幹。音訊乾淨，便宜工具也能測出價值；音訊混雜，企業級工具也會不穩。

現場字幕的第一原則：

所有會說話的人
  -> 都要用麥克風
  -> 麥克風都要進混音器
  -> 混音器分一條乾淨人聲給字幕電腦
  -> 字幕電腦送進人工智慧字幕 / 翻譯工具
  -> 觀眾掃 QR 碼在手機上看字幕

不要把筆電或手機放在會場角落聽喇叭。它會收到冷氣、回音、椅子聲、咳嗽、掌聲和喇叭延遲，字幕品質會大幅下降。

3. ADALS 要解決的不是一件事，而是六塊積木

本章重點

把現場字幕拆成麥克風、乾淨聲音、字幕電腦、工具、觀眾入口、人力修正六塊，才容易放大或縮小。

把整套現場字幕拆成六塊，事情會清楚很多。

積木 白話意思 最小版本 正式版本
A. 麥克風 誰說話，誰就要被清楚收音 主講者用手持或領夾麥 主講者、主持、座談、問答全部有麥
B. 乾淨聲音 從混音器分一條只含人聲的線 一條 AUX / REC OUT 獨立 AUX / MATRIX OUT，必要時 Dante
C. 字幕電腦 接收聲音並送進工具 一台筆電 + USB 音訊介面 主機 + 備機 + 操作員
D. 字幕工具 把聲音變成字幕 / 翻譯 Akkadu / Microsoft Translator / UDTalk 試跑 Vurbo.ai / KKCO / EventCAT / Wordly / AI-Media
E. 觀眾入口 讓觀眾掃 QR 碼看字幕 一張投影片 QR 場外海報、椅背卡、簡訊、節目手冊

4. 為什麼「混音器乾淨人聲訊號」是主角

本章重點

人工智慧不是在聽現場空氣，而是在吃一條音訊；這條音訊越乾淨，字幕越穩。

4.1 用水管來想

現場會有兩種聲音：

  • 給人聽的聲音：從喇叭放出來，讓全場聽見。
  • 給機器聽的聲音：從混音器直接拉出來，讓字幕系統聽見。

人工智慧字幕需要第二種：近、乾淨、直接。

4.2 不要讓字幕系統聽喇叭

筆電麥克風或手機麥克風聽喇叭，會碰到這些問題：

  • 聲音已經被喇叭和房間「煮過一次」。
  • 大廳堂有回音，字會黏在一起。
  • 冷氣、椅子、掌聲、咳嗽都會混進去。
  • 講者離喇叭遠近不同，音量忽大忽小。
  • 座談時多人輪流說話，人工智慧很難分辨誰在說。
  • 問答時觀眾如果不用麥克風，幾乎等於消失。

這種方式可作臨時測試，不適合正式流程。

4.3 要跟場地 / 影音團隊說的關鍵字

主辦不必懂所有設備名詞，但要知道該問什麼。

請問場地或影音團隊：

我們需要從現場混音器提供一條只含人聲的乾淨訊號給字幕電腦。可以用 AUX OUT、MATRIX OUT、REC OUT、USB 音訊介面，或你們現場可提供的等效方式。這條訊號要包含所有講者、主持人、座談和問答麥克風，但不要包含背景音樂、影片音效或現場喇叭回音。

中文白話版：

請給字幕電腦一條從混音器直接出來的乾淨人聲，不要讓電腦去聽會場喇叭。

第一段給影音團隊，第二段給場地業務或活動窗口。

5. 不同規模怎麼小型化

本章重點

同一套架構可從 20 人小測試放大到 500 人正式會議，差別在每塊積木的厚度。

這套系統可以縮小。小型活動先測流程，正式活動再加厚設備、人力與備援。

5.1 XS：20-50 人，小型工作坊

適合：內部測試、志工訓練、小型旁會。

做法：

  • 一支手持麥或會議室麥克風。
  • 一台筆電接聲音。
  • 使用 Akkadu、Microsoft Translator、UDTalk、YYSystem 這類低成本工具試跑。
  • 觀眾掃 QR 碼或開網址。
  • 一位志工看字幕是否大致可用。

風險：

  • 這不是正式無障礙服務。
  • 如果房間吵，效果會變差。
  • 適合測流程，不適合承諾品質。

5.2 S：50-150 人，低成本試跑

適合：小型公開活動、ADALS 前導場、贊助商說明會。

做法：

  • 所有台上講者都用麥克風。
  • 從混音器拉一條乾淨聲音給字幕電腦。
  • 使用 Akkadu / EventCAT / Vurbo.ai 試用或低階方案。
  • 一位字幕操作員。
  • 一位志工修正專有名詞。
  • QR 碼放投影片和報到桌。

重點：

  • 這一級開始，就不要讓筆電聽喇叭。
  • 要開始建立術語表，例如 ADALS、a11y、ARIA、WCAG、身心障礙、合理調整、台日韓講者姓名。

5.3 M：150-300 人，ADALS 正式建議級

適合：台北大型實體場地，一天或半天正式活動。

做法：

  • 主持、講者、座談、問答都有麥克風。
  • 影音團隊提供專用乾淨人聲訊號。
  • 字幕工具優先測 Vurbo.ai / KKCO；同時把 EventCAT 當價格透明備援；Akkadu 當最低成本備援。
  • 至少一位字幕操作員。
  • 至少一至兩位志工做術語表 / 專有名詞修正。
  • 至少一位問答麥克風遞送人員，確保觀眾問題也進麥克風。
  • 場內 Wi-Fi 要能支援觀眾手機開字幕頁。

5.4 L：300-500+ 人，正式國際會議

適合：國際嘉賓多、媒體或政府參與、已知有聾人 / 聽覺障礙者無障礙需求。

做法：

  • 使用 Wordly / AI-Media / StreamText / 3Play 等正式供應商，或搭配 CART 即時人工字幕。
  • 字幕與翻譯不只靠人工智慧，要有人力監看或專業字幕員。
  • 主機與備機分開。
  • 網路有備援。
  • 現場有明確服務台。
  • 如果有明確無障礙需求，人工智慧不能當唯一服務。

6. 無障礙基準服務、便利補充功能、實驗功能，要分開說

本章重點

人工智慧字幕很好用，但不等於正式無障礙服務；有明確需求時要補 CART、手語或專業字幕。

這一段影響風險、預算和對外承諾。

6.1 無障礙基準服務

意思是：如果有人需要這個服務才能平等參與，主辦要提供可靠服務。

例子：

  • CART 即時人工字幕 / 同步聽打。
  • 手語翻譯。
  • 專業字幕服務。
  • 已測過、有人力監看的字幕服務。

如果 ADALS 已經知道有聾人 / 聽覺障礙參與者，不能只說「我們有人工智慧字幕，所以可以了」。

6.2 便利補充功能

意思是：很多人會覺得好用，但它不是正式保證。

例子：

  • 觀眾掃 QR 碼看人工智慧即時翻譯。
  • 外國觀眾看繁中 / 日文 / 韓文機器翻譯。
  • 聽力正常但想看字幕的人用手機跟讀。

這很有價值。只是不要把它說成保證無障礙。

6.3 實驗試跑

意思是：我們想試，可能很好，也可能出錯，所以要有備案。

例子：

  • 第一次用 Akkadu 或 EventCAT 做台北大型實體活動。
  • 第一次讓志工即時修正人工智慧字幕。
  • 第一次把術語表放進工具。

實驗可以做，但要清楚標示，並準備備案。

6.4 不可作為唯一方案

這些做法不要當唯一正式方案：

  • 筆電內建麥克風聽喇叭。
  • 手機放講台旁邊收音。
  • 沒有彩排就直接開人工智慧字幕。
  • 沒有問答麥克風，期待人工智慧聽到觀眾喊話。
  • 沒有術語表，期待人工智慧自己懂 ADALS 和 a11y 術語。

7. 現場音訊怎麼接：用圖看就好

本章重點

所有講話都要進麥克風，再從混音器送給字幕電腦；問答麥克風是最容易漏掉的洞。

7.1 最小可用接法

講者麥克風
  -> 場地混音器
  -> 一條乾淨人聲輸出
  -> USB 音訊介面
  -> 字幕筆電
  -> 人工智慧字幕工具
  -> QR 碼
  -> 觀眾手機

這是 ADALS 最應該記住的版本。

7.2 正式宴會廳接法

講者 頭戴麥克風 / 領夾麥克風 / 手持麥克風
主持 手持麥克風
座談桌上麥或手持麥
問答無線手持麥
  -> 場地 / 外包影音混音器
  -> 現場喇叭
  -> 投影 / LED
  -> 獨立 AUX / MATRIX 乾淨人聲訊號
  -> USB 音訊介面
  -> 字幕主機
  -> 人工智慧字幕 / 翻譯平台
  -> QR 碼 / 網址
  -> 觀眾手機 / 平板 / 筆電

7.3 有備機的接法

混音器乾淨人聲訊號
  -> 音訊分配器 或 混音器第二路輸出
      -> 字幕主機
      -> 字幕備機

主機出問題：換 QR 碼 / 網址 或切備用字幕工具

如果預算有限，不一定要一開始就做備機。但正式國際會議最好有。

7.4 問答是最容易漏掉的洞

很多活動前半段字幕很好，問答一開始就崩。原因不是工具突然變笨，而是觀眾沒有進麥克風。

規則要很簡單：

  • 觀眾發問前，麥克風遞送人員把麥克風送到手上。
  • 主持人提醒：「為了字幕和線上紀錄，請等麥克風到再發問。」
  • 如果觀眾直接喊，主持人要重述問題。
  • 問答麥克風也要進混音器，也要進字幕訊號。

字幕系統只會處理進入混音器的聲音；問答麥克風漏掉，字幕就會漏掉。

  3. 餐飲：是否含午餐與茶點？可否外叫？若不可，最低消費或人頭價是多少？
  4. 影音：麥克風、混音器、投影、技術人員是否另計？
  5. 字幕：能否給字幕電腦一條只含人聲的乾淨訊號？
  6. 網路：能否支援 150-300 位觀眾同時開字幕頁？
  7. 場布：能否放 QR 碼海報、報到桌、字幕服務台？
  8. 無障礙：輪椅席、無障礙廁所、電梯、動線是否足夠？
  9. 第三方廠商：是否允許外部字幕、影音或網路廠商進場？

ADALS 的低成本方向不是只找「最便宜場地」，而是找「總成本清楚、乾淨人聲能接出來、網路可用、現場團隊願意配合彩排」的場地。飯店可以繼續保留；非飯店場地也應該同時詢問，讓決策不是被第一份飯店人頭價綁住。

14. 現場操作流程

本章重點

開場前 90 分鐘的測試、主持提醒、字幕操作員、志工修正和問答麥克風遞送人員都要有固定動作。

14.1 開場前 90 分鐘

  1. 字幕操作員到場。
  2. 找到影音技術負責人。
  3. 字幕筆電接電源。
  4. 接 USB 音訊介面。
  5. 影音團隊給乾淨人聲訊號。
  6. 主持麥測試。
  7. 講者麥測試。
  8. 座談麥測試。
  9. 問答麥測試。
  10. 開字幕工具。
  11. 掃 QR 碼測觀眾端。
  12. 用手機行動網路測一次。
  13. 用場內 Wi-Fi 測一次。
  14. 備用 QR 碼放在可找到的位置。

14.2 開場前 15 分鐘

主持人或場務提醒：

今天現場提供人工智慧即時字幕與多語翻譯。請掃描螢幕或報到桌上的 QR 碼，在手機上選擇需要的字幕語言。為了讓字幕正確，所有發言請使用麥克風，問答也請等麥克風到再提問。

這是系統規則，不只是開場提醒。

14.3 字幕操作員操作流程

操作員要看四件事：

  1. 聲音有沒有進來。
  2. 字幕有沒有持續產生。
  3. 延遲有沒有突然變長。
  4. 觀眾端是否能正常看到。

如果字幕錯幾個字，不要慌。
如果字幕完全停止，才是緊急。

14.4 志工修正操作流程

志工不需要修每個逗號。
志工要優先修會造成理解錯誤的東西：

  • 講者姓名。
  • 組織名稱。
  • ADALS。
  • WCAG、ARIA、a11y。
  • 地名。
  • 政策或法規名。
  • 數字。
  • 時間。
  • 網址。

志工的任務是修正會影響理解的錯誤，不是追求逐字完美。

14.5 問答麥克風遞送人員操作流程

麥克風遞送人員的任務不是遞麥克風而已。
麥克風遞送人員是字幕系統的守門員。

規則：

  • 看到觀眾舉手，立刻走近。
  • 麥克風到觀眾手上後才讓他說。
  • 如果觀眾已經開始喊，主持人要請他等麥克風。
  • 如果問題很長，主持人重述一次。
  • 不要讓多人同時搶話。

問答要慢一點，確保每個問題都進麥克風。

15. 壞掉時怎麼救

本章重點

先判斷是沒聲音、爆音、延遲、Wi-Fi、QR、沒拿麥克風，然後用對應流程救回來。

15.1 沒聲音

可能原因：

  • 混音器沒有送訊號。
  • 線沒插好。
  • USB 音訊介面沒被電腦選到。
  • 字幕工具選錯輸入。

處理：

  1. 操作員看電腦輸入音量表。
  2. 影音團隊看混音器輸出。
  3. 換一條線。
  4. 重選音訊輸入。
  5. 必要時切到備援工具。

15.2 聲音爆掉

症狀：

  • 字幕突然亂掉。
  • 聲音音量表一直紅色。
  • 人工智慧把一句話切成奇怪的字。

處理：

  • 請影音團隊降低訊號音量。
  • 不要只調現場喇叭音量。
  • 重新測主持人最大音量。

爆音會讓辨識品質下降，不會讓字幕更清楚。

15.3 字幕延遲

可能原因：

  • 網路慢。
  • 工具伺服器慢。
  • 翻譯語言太多。
  • 電腦太忙。

處理：

  • 減少不必要語言。
  • 關掉電腦其他程式。
  • 切 5G 熱點測試。
  • 告知主持人放慢速度。
  • 如果延遲仍高，切備援。

15.4 Wi-Fi 壞掉

處理：

  • 場內投影 QR 碼旁邊放備用說明。
  • 提醒觀眾可用自己手機網路。
  • 操作員改用 5G 熱點。
  • 如果觀眾端都不能用，改投影單一語言字幕到螢幕。

15.5 QR 碼掃不到

處理：

  • 放大 QR 碼。
  • 提供短網址。
  • 報到桌放紙本。
  • 投影片停留久一點。
  • 不要把 QR 碼放太小或放在角落。

15.6 講者沒拿麥克風

處理：

  • 主持人立即提醒。
  • 影音團隊不要期待字幕系統收得到沒進混音器的聲音。
  • 如果講者短暫離麥，主持人重述重點。

15.7 人工智慧翻錯專有名詞

處理：

  • 先修術語表。
  • 志工優先修高頻詞。
  • 主持人可在開場先念正確讀法。
  • 會後把錯詞加到下一場術語表。

16. 術語表：這是讓人工智慧少犯傻的字典

本章重點

ADALS、a11y、WCAG、ARIA、人名與組織名要先準備，人工智慧才比較不會亂猜。

如果沒有術語表，人工智慧會把不熟的詞硬猜成常見詞。
例如：

  • ADALS 可能被聽成其他英文縮寫。
  • a11y 可能被轉成無障礙或奇怪字串。
  • WCAG 可能被拆錯。
  • 台灣人名、日文人名、韓文人名可能被亂翻。

ADALS 應在活動前準備：

  • 活動名稱。
  • 主辦單位。
  • 講者姓名。
  • 講者公司 / 組織。
  • 專有名詞。
  • 常見縮寫。
  • 地名。
  • 法規 / 標準名稱。
  • 產品名稱。
  • 贊助商名稱。

術語表不用一開始完美，但要在每次測試後更新。

17. 資料與隱私

本章重點

把聲音送進字幕工具就像交給外部打字員，要確認保存、刪除、訓練與資料所在地。

現場字幕會把聲音送到工具服務。這代表主辦要問清楚：

  • 聲音會不會被保存？
  • 逐字稿會不會被保存？
  • 誰可以下載？
  • 會保存多久？
  • 能不能刪除？
  • 是否用來訓練模型？
  • 資料存在哪個國家？
  • 有沒有資料處理協議或隱私條款？

如果活動包含敏感內容，請不要只看價格。
便宜工具可以用在公開活動；敏感內容要更小心。

把聲音送進字幕工具前，要確認資料是否保存、轉交、下載或用於模型訓練。

18. 最終建議

本章重點

先試測、台灣優先、EventCAT 備援、Akkadu 低成本備援；有正式無障礙需求就加人工服務。

ADALS 最務實的路線是：

  1. 先做一場小型試測，不要直接採購正式方案。
  2. 試測必須用混音器乾淨人聲訊號，不要用筆電聽喇叭。
  3. 台灣優先測 Vurbo.ai 和 KKCO。
  4. 同時測 EventCAT，因為它價格和會議流程比較透明。
  5. 測 Akkadu，作為低成本和備援工具。
  6. UDTalk 作為 A11y Tokyo 類型的人力修正流程參考。
  7. 若有正式無障礙需求，另找 CART / 專業字幕服務 / 手語，不要只靠人工智慧。

一句話版本：

用台灣工具優先，但不要把希望押在單一應用程式。先把聲音管線做好，再比較工具。正式場用正式建議方案；高風險場加人工字幕和企業級供應商。

最推薦的正式組合：

場地 / 外包影音乾淨人聲訊號
  + Vurbo.ai 或 KKCO 作台灣優先主方案
  + EventCAT 作價格透明備援
  + Akkadu 作低成本 緊急 備援
  + 志工術語表 / 專有名詞修正
  + 問答麥克風遞送人員

這套組合的重點是可執行、可備援、非技術人員能接手。
